# P4-01 MTGO Front-end Split Audit

## Purpose

P4-01 splits the monolithic MTGO page into maintainable static assets while preserving the existing Standard product. It does not redesign the page, change statistics, enable another format, expose subtypes, or add Melee behavior.

## Static asset split

The 1,225-line `index.html` is now a 65-line HTML shell. It loads:

- `assets/css/site.css`, containing the existing stylesheet in its original order;
- `assets/js/common.js`, containing browser helpers that do not own MTGO product state;
- `assets/js/mtgo.js`, containing localization data, MTGO state, data loading, rendering, event handling, and initialization.

Chart.js remains the first script. `common.js` and `mtgo.js` are classic scripts loaded at the original end-of-body initialization point. This preserves global `setLang(...)` access for the existing language buttons and avoids introducing a module loader, bundler, framework, package manifest, or build step.

The existing Standard public contract now searches the HTML and both JavaScript assets for every frozen public data-path template. A new focused test also protects:

- the exact stylesheet path;
- Chart.js, common-helper, and MTGO script order;
- the absence of inline style and script blocks;
- the absence of `type="module"`;
- asset presence;
- the materially smaller HTML shell;
- classic language hooks and final `refreshAll()` initialization.

The repository validator follows the same three front-end sources when checking those templates, and the Phase 3 hardcoded-inventory contract now tracks the unchanged Standard format boundary at `assets/js/mtgo.js` instead of its former inline location in `index.html`.

During implementation, an initial mechanical extraction exposed that the data-path contract alone could pass malformed HTML. The generated files were reset before continuing, and the focused structural test was added so that markup and asset-loading corruption fails automatically.

## Preserved behavior

The repository root was served over local HTTP and the split page was exercised in the in-app browser. The following passed with zero console errors or warnings:

- the post-split default state matched the pre-split title, language, active view, visible range buttons, one-week row count, and labels;
- stylesheet loading moved from inline CSS to `assets/css/site.css`;
- scripts loaded in the order Chart.js, `common.js`, and `mtgo.js`;
- English and Chinese localization remained functional;
- four-week statistics loaded 1,056 decks and 29 archetype rows;
- the Izzet Prowess deck-detail row expanded with representative-deck and deviation content;
- the four-week matchup view rendered 38 rows and 1,482 matrix cells;
- Weekly Pickup loaded `2026-W27` and both report groups;
- a 390-by-844 viewport retained the existing navigation, range controls, chart, and responsive behavior.

The visible ranges remain 1, 4, and 12 weeks. Generated 36-week documents remain public but intentionally absent from the page selectors. The existing optional-data fallback code is unchanged.
