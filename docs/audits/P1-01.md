## 2. Executive summary

The repository contains a working, Standard-focused MTGO data pipeline with fetch, classification, rolling statistics, Videre matchup, Weekly Pickup, metadata, and GitHub Pages entry points. The audit statically mapped those paths and validated existing data syntax. It did not execute the production pipeline, fetch data, or establish reproducibility.

The principal Phase 1 gaps are absent dependency manifests, tests, schemas, baseline fixtures, PR validation, and overlapping automated data workflows. Rule matching currently uses first-match order without explicit rule IDs, priorities, or conflict reports.

There are 14 tracked root Python scripts, 735 tracked JSON files, 6 tracked YAML/YML files, one monolithic `index.html`, and authoritative documentation under `docs/`. Generated event and statistics JSON are committed. `MTGOFormatData/`, downloaded HTML, `__pycache__/`, and several diagnostic outputs are ignored/local-only. There is no `README.md`, `LICENSE`, `NOTICE.md`, requirements manifest, `tests/`, `schemas/`, or pytest configuration. No Git tags exist.

## 4. Production entry points and command chain

- `batch_mtgo.py`: network fetcher for all listed MTGO formats; reads/writes `fetched.txt` and `data/<format>/`.
- `classify_standard.py`: reads `data/standard/` and `my_archetypes/standard.yaml`; writes ignored `unknown_decks.txt`; unmatched decks are `Unknown`.
- `stats_standard.py`: writes `stats/standard/mtgo/range_{1,4,12,36}w.json`, `decks_{1,4,12,36}w.json`, and `index.json`.
- `fetch_videre_matches.py`: fetches Videre matches into `data/standard/mtgo/matches/`.
- `stats_matchup.py`: writes Standard matchup windows and indexes.
- `dump_unknown_highperf.py`: writes `unknown_highperf.txt`.
- `cluster_unknown.py`: writes `unknown_clusters.txt`.
- `weekly_pickup.py`: generates candidate YAML and published Weekly Pickup JSON/index files.
- `gen_meta.py`: writes `stats/standard/mtgo/meta.json` using rules and Git metadata.
- `event_report.py`, `fetch_mtgo.py`, `verify_videre.py`, `recon.py`, and `recon_list.py` are diagnostic or legacy entry points not invoked by current workflows.

The `update.yml` chain is: fetch MTGO, fetch Videre, build Standard stats, build matchup, dump/cluster Unknowns, conditionally generate Weekly Pickup candidates, generate metadata, then commit/push. `scrape.yml` runs only the MTGO fetcher and commits `data/` plus `fetched.txt`.

## 6. GitHub Actions audit

### `scrape.yml`

It is named `Daily MTGO Scraper`, runs daily at `04:00 UTC`, and supports `workflow_dispatch`. It uses Python 3.12, installs unpinned `requests`, runs `batch_mtgo.py`, then stages `data/` and `fetched.txt` and pushes. It has no explicit permissions block and no concurrency configuration. No named repository secret is explicitly referenced; `actions/checkout` normally provides persisted Git authentication for later Git commands. Effective write permission depends on repository or organization defaults because this workflow has no explicit permissions block.

### `update.yml`

It is named `Weekly data update`, but its cron schedule currently executes daily at `20:00 UTC` (05:00 JST the following day). It supports `workflow_dispatch`, uses Python 3.11, explicitly grants `contents: write`, and has concurrency group `data-update` with `cancel-in-progress: false`. It installs unpinned `requests pyyaml`, runs the complete production chain, and commits/pushes generated data and reports. Diagnostic Unknown and candidate steps use `continue-on-error`.

Both workflows invoke `batch_mtgo.py` and overlap on `data/` and `fetched.txt`. Only `update.yml` has explicit concurrency, so manual or scheduled runs can race across workflows. Both use implicit target branches and `git pull --rebase` before pushing. No named secrets are referenced. No workflow summary/reporting step was found.

Recent automation commit `11e5bdf` changed generated Modern/Vintage data, Standard statistics, `fetched.txt`, and Unknown output, confirming that generated data is committed.
