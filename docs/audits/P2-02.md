# P2-02 Shared Card-Name and Deck Normalization Audit

## Purpose

P2-02 extracts format- and source-independent card-name and deck-count normalization from the legacy Standard classifier while preserving the approved parent-archetype baseline.

## Shared implementation

The shared package now provides:

- `src/mtgmeta/card_names.py`: whitespace normalization and explicitly maintained printed-name aliases;
- `src/mtgmeta/deck.py`: main-deck and sideboard count aggregation plus zone-aware card counting;
- `src/mtgmeta/data/om1_spm_aliases.json`: a traceable OM1 printed-name to SPM canonical-name mapping;
- `src/mtgmeta/__init__.py`: the public shared utility surface.

`classify_standard.py` retains its existing public helper names. Its deck-count path delegates to the shared implementation, while the legacy `normalize_name` helper keeps the original two-alias behavior because `stats_standard.py` still uses that helper for public average-deck and deviation output. Migrating that consumer would change public deck names and one observed deviation value, so it is deliberately deferred to a separately reviewed integration task. Rule loading, signature matching, first-match selection, reporting, statistics, schemas, workflows, and front-end behavior are not migrated in this task.

## OM1 and SPM mapping research

The checked-in artifact was derived from the public [MTGJSON OM1 dataset](https://mtgjson.com/api/v5/OM1.json), version `5.3.0+20260718`, dated `2026-07-18`. MTGJSON describes its project and data as free and open source under the [MIT License](https://mtgjson.com/faq/). Wizards' release notes independently identify `SPM` as the Marvel's Spider-Man set code and describe its Constructed legality.

MTGJSON represents each OM1 printing with the SPM rules name in `name`, the OM1 visible name in `printedName`, and a shared Scryfall oracle ID. Selecting records where those names differ produced:

```text
unique_aliases=158
unique_canonical_names=153
```

The five-name difference comes from five transforming cards whose two OM1 face names map to one canonical `front // back` name. Every mapping retains its oracle ID and collector number. Representative mappings include:

- `Kavaero, Mind-Bitten` → `Superior Spider-Man`;
- `Leyline Weaver` → `Spider Manifestation`;
- `Surris, Silk-Tech Vanguard` and `Surris, Spidersilk Innovator` → `Peter Parker // Amazing Spider-Man`.

The runtime does not fetch MTGJSON. The reviewed artifact is deterministic and available offline.

## Compatibility results

The complete mapping changes normalized card-count dictionaries for 567 of the 3,936 frozen Standard decks and covers 25 alias names observed in that corpus. This is expected: the legacy implementation normalized only two aliases.

All 3,936 parent-archetype results remain identical to the Phase 1 baseline:

```text
records=3936
normalization_changed_decks=567
observed_alias_names=25
parent_archetype_differences=0
unknown=71
multiple_matches=947
```

No archetype, subtype, or rule definition changed.

The three Standard generators were also run in matched baseline and candidate temporary copies with `PYTHONHASHSEED=0`. A field-level comparison of all generated Standard MTGO JSON, excluding only `generated`, `data_updated`, and `rules_updated` timestamps, found `semantic_field_differences=0`. The compatibility-only legacy `normalize_name` boundary therefore preserves current statistics and page-consumed data while the complete shared mapping is established for later reviewed migration.

## Validation boundaries

P2-02 must pass:

- complete artifact provenance, count, uniqueness, and identifier checks;
- representative single-faced and transforming-card alias checks;
- duplicate aggregation, zone counting, quantity coercion, whitespace, empty-deck, and input-immutability checks;
- full frozen-corpus parent compatibility;
- the complete repository test suite and validators.

P2-02 does not change generated statistics, public JSON, schemas, workflows, or front-end files. P2-03 remains unauthorized until P2-02 is accepted and published.

The project owner accepted P2-02 and authorized its publication on 2026-07-19. P2-03 remains subject to separate owner authorization after publication.
