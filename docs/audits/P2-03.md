# P2-03 Shared Rule Model, Loader, Validation, and Schema Audit

## Purpose

P2-03 defines the versioned configuration boundary required before Standard rules can be migrated. It adds shared immutable rule models, safe YAML loading, semantic validation, and a Draft 2020-12 JSON Schema without changing the production Standard rule file or classification behavior.

## Implemented contract

Version `1.0.0` models:

- a format-specific `RuleSet`;
- stable archetype IDs, display names, and explicit priorities;
- optional subtype IDs nested beneath one parent archetype;
- globally unique rule IDs and explicit rule priorities;
- nullable rule-to-subtype references;
- all-of card conditions with `main`, `side`, or `any` zones;
- minimum, maximum, exact, and default-at-least-one counts.

Models are immutable dataclasses. The loader returns a model only after validation succeeds.

## Validation layers

`schemas/classification-rules.schema.json` defines the machine-readable shape and rejects missing required fields, malformed IDs, invalid scalar types, unsupported zones, empty collections, unknown properties, and incompatible exact/range count fields.

The shared Python validator adds relationship checks that JSON Schema cannot conveniently express:

- duplicate archetype IDs;
- globally duplicate rule IDs;
- duplicate subtype IDs within one parent;
- rule references to missing or cross-parent subtypes;
- declared subtypes that are not associated with any rule;
- unsupported schema versions;
- invalid priorities, including booleans;
- minimum counts greater than maximum counts.

Duplicate YAML keys are rejected before model construction. Errors retain deterministic paths such as `archetypes[0].rules[0].subtype_id`.

Equal priorities are intentionally valid configuration. Whether two equal-priority rules conflict depends on a deck matching both rules and belongs to P2-05 full-match evaluation.

## Legacy boundary

`validate_rules.py` automatically recognizes a versioned rule file by `schema_version`. Without that field it retains the existing legacy validation path. Therefore the default command still validates `my_archetypes/standard.yaml` unchanged, while an explicit versioned fixture exercises the shared contract.

P2-03 does not migrate any of the 76 Standard rules. That migration remains P2-04.

## Validation evidence

- focused model and validator tests cover valid models, equal priorities, subtype references, duplicate IDs and YAML keys, invalid versions and IDs, invalid priority and count types, unsupported fields and zones, empty rules, condition bounds, unreadable inputs, CLI errors, and JSON Schema behavior;
- all repository schemas are checked as Draft 2020-12 and version `1.0.0`;
- the legacy Standard rule validator remains read-only and passing;
- the 3,936-record Standard quality baseline remains `unknown=71` and `multiple_matches=947`;
- production rules, classification, statistics, public JSON, workflows, and front-end files are unchanged.

Final validation passed with `130` pytest tests. The three Standard generators were run in matched baseline and candidate temporary copies with `PYTHONHASHSEED=0`; a field-level comparison of every generated Standard MTGO JSON file, excluding only `generated`, `data_updated`, and `rules_updated`, returned `semantic_field_differences=0`.

P2-04 remains unauthorized until P2-03 is accepted and published.

The project owner accepted P2-03 and authorized its publication on 2026-07-20. P2-04 remains subject to separate owner authorization after publication.
