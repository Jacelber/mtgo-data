# P2-07 Classification Diagnostic Reports Audit

## Purpose

P2-07 converts the full-match diagnostics introduced in P2-05 and connected to production in P2-06 into deterministic, versioned, machine-readable Standard MTGO reports. The task does not change classification rules, parent-archetype selection, statistical formulas, public page data, or front-end behavior.

## Scope and path resolution

The live status entry initially listed `stats/standard/` as a target. `docs/DATA_ARCHITECTURE.md`, which has higher authority, assigns human-review and machine-readable quality reports to `reports/<format>/<source>/`. P2-07 therefore writes to `reports/standard/mtgo/` and corrects the status target without changing product scope.

## Generated reports

The generator writes six Schema-versioned documents:

- `index.json`: full-corpus counts, strict-validation status, and report catalog;
- `unknown_decks.json`: valid unclassified decks with normalized main-deck and sideboard evidence;
- `multiple_matches.json`: all matching rules and the deterministic selected result;
- `classification_conflicts.json`: unresolved top-priority conflicts and blocking status;
- `overridden_matches.json`: lower-priority matches retained after selection;
- `subtype_diagnostics.json`: selected compatibility subtypes and all subtype candidate overlaps.

All documents declare Standard, MTGO, and `all_available_events` scope. The generator derives `data_through` from the input events instead of adding a wall-clock timestamp, so unchanged inputs produce byte-identical output.

## De-identification boundary

The reports omit player names, MTGO login IDs, raw player IDs, usernames, and complete source player records. Each record receives a stable 20-character pseudonymous deck ID derived from event context, record position, and normalized deck contents without using a player identity.

Event ID, event name, event time, source filename, deck evidence, rule identities, priorities, and condition evidence remain available because they are necessary for diagnosis. This is data minimization and pseudonymization, not a claim that a public tournament decklist can never be correlated with its public source.

Existing page-consumed JSON and its player fields are outside the report path and remain unchanged. The current front end does not load these reports.

## Strict validation

`generate_classification_reports.py --strict` writes the reports and then fails when an unresolved classification conflict or invalid deck input exists. Writing before failure preserves runner-local evidence for diagnosis. Unknown, deterministic multiple matches, overridden matches, and resolvable subtype overlaps do not block production.

The production update workflow runs strict report generation after statistics and metadata generation, validates all declared JSON Schemas, and stages `reports/` with other generated outputs.
