# P5-07 Melee quality gates and publication blocking

## Purpose

P5-07 turns the P5-06 normalized event into a fail-closed publication candidate.
It does not fetch Melee, write production data, classify Modern decks, calculate
statistics, alter workflows, or expose a Tabletop front end.

## Implemented boundary

`src/mtgmeta/melee/quality.py` provides two operations:

- `finalize_event_quality` deep-copies and assesses a Schema-shaped normalized
  event without mutating its input;
- `build_publication_payload` returns canonical UTF-8 JSON bytes only after the
  assessed event is publishable, otherwise it raises
  `MeleePublicationBlocked` before any output write.

The implementation performs no network request and contains no output writer.
`jsonschema` is now a runtime dependency because Schema validation is part of
the runtime publication boundary rather than only a development check.

During complete Windows regression testing, five existing byte-identical
Standard tests exposed platform-default CRLF output while committed files use
LF. A CRLF checkout was not a valid workaround because it correctly invalidated
the byte counts and SHA-256 values of raw Melee fixtures. The focused repair
therefore makes the existing MTGO statistics, matchup, Pickup, classification-
report, and Melee manifest text writers request LF explicitly. This changes no
JSON/YAML values, paths, formulas, or Linux output and makes the existing byte
contract deterministic on Windows.

## Quality rules

The gate validates the complete event against normalized Melee event Schema
2.0.0 before assessment and validates the finalized document again afterward.
It then checks:

- explicit whitelist `enabled` and `verified` authorization;
- event identity, name, date, Constructed format, series, and structure;
- presence and SHA-256 integrity metadata for raw artifacts;
- uniqueness of normalized and source identities;
- participant, phase, round, standing, decklist, and match references;
- round-to-phase semantic agreement;
- participant, decklist, and result resolution;
- played-result pairs and match points;
- Constructed-statistics and matchup eligibility against the configured format,
  Swiss state, round phase, and played result;
- presence of at least one verified played Constructed Swiss match.

Missing or unavailable decklists produce deterministic non-blocking warnings.
No numeric coverage threshold is introduced; OPEN-002 remains deferred to
Phase 8. Any blocking issue sets `status: blocked` and `publishable: false`.

## Schema 2.0.0

The major version records tightened publication semantics. It requires:

- played matches to contain two played-result competitors;
- nonplayed matches to be ineligible;
- matchup eligibility to imply Constructed-statistics eligibility and play;
- blocked status to include a blocking issue and remain non-publishable;
- any blocking issue to force blocked, non-publishable quality;
- publishable quality to contain no blocking issue;
- quality issues to be unique.

The synthetic reference contract was upgraded to 2.0.0. No public Standard
Schema or generated JSON contract changed.

## Determinism and failure behavior

Quality issues are de-duplicated and sorted by stable semantic keys. Source
evidence is de-duplicated and sorted. Publication JSON uses sorted keys, compact
deterministic separators, UTF-8, and one trailing line feed. Tests prove that
the same logical event, including a different top-level dictionary insertion
order, produces byte-identical payloads and identical SHA-256 values.

Malformed Schema input raises `MeleeQualityError`. Valid but unauthorized or
unsafe input raises `MeleePublicationBlocked`. Neither path writes a file.
