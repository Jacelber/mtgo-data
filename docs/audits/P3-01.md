# P3-01 MTGO Format Pipeline Contract Audit

## Purpose

P3-01 defines the executable migration contract for replacing Standard-only MTGO pipeline assumptions with explicit format configuration. It inventories the current coupling before production refactoring begins. The task changes no generator, workflow, classification rule, statistic, public JSON, or front-end behavior.

## Baseline

The contract is pinned to the published Phase 2 recovery baseline:

```text
tag=phase-2-shared-classifier-baseline
commit=a9e35a2c957d32f2bc0fb43c8a0777a25f71c31c
pytest=156 passed
```

Standard remains the only executable format. Pauper, Modern, Pioneer, and Legacy remain planned but non-executable. Vintage remains decision-gated and must not be enabled by pipeline generalization.

## Current coupling inventory

The executable fixture records twelve current coupling boundaries:

1. `batch_mtgo.py`: event discovery and format-directory storage;
2. `classify_standard.py`: Standard rule and event paths;
3. `fetch_videre_matches.py`: Standard Videre source and output path;
4. `stats_standard.py`: Standard event input and statistics output;
5. `stats_matchup.py`: official event, match, and output paths;
6. `weekly_pickup.py`: Standard Pickup output;
7. `gen_meta.py`: Standard rules and metadata output;
8. `generate_classification_reports.py`: Standard report defaults;
9. `src/mtgmeta/reports.py`: Standard report metadata;
10. `validate_repository.py`: Standard public-path validation;
11. `.github/workflows/update.yml`: legacy production command sequence and Pickup path;
12. `index.html`: the current front-end format-selection compatibility boundary.

The inventory intentionally includes validation and front-end boundaries even though P3-01 does not change them. A generalized producer is incomplete if its validators or public consumer still rely on an undocumented Standard-only assumption.

## Format state contract

The contract distinguishes three states:

- executable: `standard` only;
- planned but non-executable: `pauper`, `modern`, `pioneer`, and `legacy`;
- decision-gated: `vintage`.

The future generalized command must require an explicit format. Legacy wrappers may continue to select Standard, but a generalized command must never silently fall back to Standard for an unknown or disabled format.

DEC-033, accepted during P3-07, later clarified that official MTGO event archival is an independent permission rather than full product execution. This historical P3-01 executable-format baseline remains unchanged: only Standard can run product capabilities, while an explicitly collection-enabled format may archive only its own event data.

## Path isolation contract

Format configuration must control these five path families:

```text
data/{format}
data/{format}/mtgo/matches
my_archetypes/{format}.yaml
stats/{format}/mtgo
reports/{format}/mtgo
```

The Standard resolutions are tested against existing tracked directories and the Phase 1 public-contract fixture. Cross-format reads and writes are explicitly prohibited.

## Migration sequence

The contract assigns every Phase 3 capability exactly once:

- P3-02: format registry and validated path resolution;
- P3-03: MTGO event fetch, storage, normalization, and classification dispatch;
- P3-04: event and rolling-range statistics;
- P3-05: Videre matchup processing and classification-report routing;
- P3-06: Weekly Pickup, metadata, and catalogs;
- P3-07: generalized command entry point and production workflow migration;
- P3-08: fixed-reference regeneration, public-contract regression, and browser closeout.

Temporary root-level commands remain compatibility entry points until their generalized replacements, workflow callers, documentation, and regressions are verified.
