# P5-03 — Safe Melee client and raw archive

## Outcome

P5-03 adds a bounded raw-response client for the manually approved Melee whitelist. It preserves source responses in immutable snapshots and does not parse tournament data, normalize records, classify decks, generate statistics, change public JSON, change workflows, or change the front end.

The reference event `434455` remains `enabled: false`. Its disabled state is verified before the client chooses a transport, inspects an archive root, or makes a request. No live Melee request was made during this task.

## Request and archive boundary

Whitelist Schema 2.0.0 adds a required, explicit `raw_requests` plan. A request must use HTTPS, target `melee.gg`, declare its resource type, and match either the whitelisted event's `/Tournament/View/<event_id>` path or an explicit `/Decklist/View/<decklist_id>` path. The client does not discover links from source responses and does not follow redirects. Optional event-resource pagination is a bounded configured query parameter with a maximum of 100 pages. The complete plan is capped at 500 responses.

The client uses finite retries only for transient HTTP statuses and transport failures. It validates positive timeouts and retry counts plus non-negative delays. It sends no cookies or authorization headers. The CLI defaults to dry run and requires `--execute` before it can use the live transport; both modes still require an enabled, verified whitelist entry.

Raw responses are streamed under `data_raw/melee/<event_id>/<utc-snapshot>/`, with a 32 MiB per-response limit and 256 MiB per-snapshot limit. Each completed snapshot contains the original response bytes and a Schema-validated manifest recording source URL, timestamp, status, attempt count, declared and actual content types, safe `ETag` and `Last-Modified` metadata, file path, byte count, and SHA-256. Sensitive response headers are not retained. Writes first use a sibling temporary directory and publish only through an atomic rename. Failures remove the temporary directory; re-fetches allocate a new snapshot and never overwrite older evidence.
