## Workflow result

The existing official-event loop is unchanged and continues to collect:

- Standard;
- Legacy;
- Pioneer;
- Pauper;
- Vintage;
- Modern.

The complete product loop contains only Standard and Modern. For both formats it
runs:

1. Videre match collection;
2. event and rolling-range statistics;
3. matchup statistics;
4. Weekly Pickup candidate preparation with `--if-absent`;
5. metadata generation;
6. strict de-identified classification reports;
7. exact rule-file validation.

The maintained hierarchy catalog is additionally regenerated for Modern.
Standard hierarchy generation is deliberately excluded because P6-09 owns the
Standard hierarchical calculation and compatibility proof.

Weekly Pickup remains candidate-only. The workflow attempts both formats even
if one candidate command fails, retains `continue-on-error` only on this
preparation step, and never approves or publishes a candidate or changes known
state.

The existing master-only guard, concurrency group, 45-minute timeout,
least-privilege write permission, clean-checkout test gate, snapshot-before-fetch
ordering, generated-file staging scope, and remote publication confirmation are
unchanged.

## Dynamic candidate gate

The baseline snapshot schema advances from `1.0.0` to `2.0.0` because a scalar
Standard match count is replaced by per-product match counts.

The validator derives two sets from `configs/formats.yaml`:

- collection formats, using `event_collection_enabled`;
- complete products, requiring enabled MTGO execution and all seven production
  capabilities.

It then:

- records event counts for every collection format;
- records match counts for every complete product;
- rejects event or match count regressions per format;
- validates the embedded format of changed event documents;
- permits match, statistics, and report changes only for complete products;
- rejects product output for archive-only formats;
- reports change areas by format;
- continues to reject deletions, renames, conflicts, cross-product writes,
  duplicate fetched-ledger entries, malformed documents, and arbitrary new
  generated JSON paths;
- permits only expected new raw event and match archives and dated Pickup
  candidate or base-reference YAML.

An old `1.0.0` baseline fails closed instead of being silently interpreted.

## Mutable production coverage regression

The first full-suite run exposed an existing P6-07 integration-test assumption.
The test froze Modern coverage at 183 official events and 22 events without a
Videre archive. The latest committed production update added two official
events without corresponding archives, so the correct live committed values
became 185 and 24. This would have blocked the clean-checkout production gate
before P6-08 could fetch anything.

The test now independently derives official and archive event-ID sets from the
committed inputs, verifies the exact coverage document generated from those
sets, and retains the assertion that coverage is partial. It therefore continues
to detect calculation or source-routing errors without treating a daily data
increase as a code regression. The generator and committed metadata were not
changed.
