# P2-06 Legacy Standard Entry-Point Integration Audit

## Purpose

P2-06 routes the existing Standard production and support entry points through the P2-05 full-match shared classifier while preserving the parent-archetype string interface consumed by current statistics, matchup generation, Weekly Pickup, and front-end data.

This task changes the internal decision path. It does not publish subtype fields, change statistical formulas, or generate P2-07 diagnostic artifacts.

## Compatibility architecture

`LegacyArchetypeRules` remains a list-compatible view of the 76 migrated rules and now retains the versioned `RuleSet` that produced it. Existing callers can continue to iterate the legacy list shape, while the Standard compatibility wrapper uses the attached `RuleSet` for classification.

`classify_standard.py` adds:

- `classify_standard_result()` for the complete P2-05 `ClassificationResult`;
- `all_matching_archetype_names()` for shared full-match quality analysis;
- a `match_archetype()` compatibility wrapper that returns the existing parent display name or `None`;
- explicit `StandardClassificationConflict` and `StandardInvalidDeck` failures.

For a valid current Standard deck, the wrapper returns the same value as before. Subtype, selected rule, all matches, and overridden matches remain available in the full result without changing the legacy string result.

Conflicts and invalid decks are no longer silently compressible into Unknown. The compatibility wrapper raises a deterministic exception containing sanitized rule identities and retains the complete shared result on the exception object.

Plain external legacy rule lists remain supported by the old matching fallback. Repository production rules loaded through `load_rules()` always carry the shared `RuleSet` and therefore use the shared classifier.

## Production and support consumers

`stats_standard.py` and `stats_matchup.py` already import the central `load_rules()`, `deck_to_counts()`, and `match_archetype()` functions. Their production classification path therefore switched to P2-05 without changing their statistical or matchup code.

`weekly_pickup.py` continues to call `stats_standard.process_event()` and receives the same parent archetype grouping.

Copied legacy matching loops were removed from:

- `event_report.py`;
- `cluster_unknown.py`;
- `dump_unknown_highperf.py`.

`validate_standard_quality.py` now uses the shared all-match helper while retaining the frozen aggregate quality contract.

Integration tests verify that Standard statistics, matchup code, and auxiliary tools reference the central compatibility function, and that a `stats_standard.process_event()` classification invokes the shared classifier.

## Boundaries and next step

P2-06 does not change Standard rules, source data, schemas, workflows, public statistics, front-end files, or subtype presentation. The list-shaped compatibility adapter remains available while legacy consumers still exist, but it no longer makes production classification decisions.

P2-07 must not begin until the project owner accepts P2-06, P2-06 is published, and P2-07 receives separate authorization.

The project owner accepted P2-06 and authorized pull-request creation and merge on 2026-07-20. P2-07 remains subject to separate task authorization after publication.
