# P5-02 — Melee whitelist loading and fetch-boundary enforcement

## Outcome

P5-02 turns the manually maintained Melee whitelist into an immutable, fail-closed runtime registry. It performs no network request, raw-data collection, normalization, classification, statistics generation, or front-end change.

The loader reads `configs/melee_events.yaml` with duplicate-key rejection, validates its source-independent semantic rules, and exposes two intentionally different operations:

- `get(event_id)` inspects a known event without authorizing collection;
- `require_fetchable(event_id)` returns an event only when it is explicitly enabled and verified.

The registered Modern reference event `434455` remains `enabled: false`. It can be inspected but is rejected by `require_fetchable()`.

## Validation boundary

The loader rejects malformed configuration before a future client can receive an event URL. Checks include:

- duplicate YAML keys and duplicate event IDs;
- malformed Melee URLs and URL/ID mismatches;
- invalid or reversed dates;
- unsupported series, structures, formats, and review states;
- enabled events that are not verified;
- team or non-tabletop entries;
- mixed-structure mismatches;
- duplicate phase IDs, round assignments, or source labels;
- Draft phases that are not Limited;
- Constructed phases or statistics settings that disagree with the declared Constructed format;
- unknown phase assignments in enabled events;
- any attempt to fetch an unlisted or disabled event.

The data classes use frozen dataclasses and tuples so a caller cannot mutate a validated event into an enabled state after loading.
